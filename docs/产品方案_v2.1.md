## 一、产品定位与目标

### 1.1 一句话定义

> 一个自动采集 Git 提交记录 + 用户自定义文档，基于 LLM 提取结构化事实并通过模板引擎渲染，按周生成结构化工作周报的 CLI/Web 双模 Agent。

### 1.2 核心痛点

| 痛点 | 现状 | 本产品解决方式 |
|------|------|----------------|
| 周五赶周报，回忆遗漏 | 手动翻 Git log、翻聊天记录 | 自动采集 Git，按天聚合 |
| 写周报耗时 30-60min | 手动组织语言、排版 | LLM 提取事实 + 模板自动渲染 |
| 个人笔记/文档散落各处 | 本地 .md/.doc 无统一管理 | 统一摄入，语义检索 |
| 周报质量不稳定 | 状态好写得好，累了就敷衍 | Agent 保证结构一致、数据完整 |

### 1.3 目标用户

- **主力用户**：软件工程师、研发团队 TL
- **扩展用户**：产品经理、设计师、数据分析师（任何需要周报 + 有 Git 或文档工作流的人）

### 1.4 成功指标

| 指标 | 目标值 |
|------|--------|
| 周报生成时间 | < 2 分钟（含用户审阅） |
| 用户手动修改率 | < 20%（生成后直接可用） |
| Git 日志覆盖率 | 100%（所有 commit 不遗漏） |
| 用户满意度 | > 4.0 / 5.0 |

## 三、核心生成流程：结构化输出 + 模板渲染（V2.0 关键修订）

### 3.0 设计决策：为什么选路径 B

V1.0 存在两条冲突路径：
- **路径 A（LLM 直接写 Markdown）**：模板系统形同虚设
- **路径 B（LLM 输出结构化数据 + 模板渲染）**：可维护、可控、多格式导出

**V2.0 选定路径 B**，核心变化：

```
V1.0:  LLM ──(Markdown)──► write_report ──► 文件
V2.0:  LLM ──(JSON)──► Schema 校验 ──► Jinja2 渲染 ──► 文件
```

Agent 的目标从"撰写 Markdown 周报"改为"**提取并组织周报所需的结构化事实**"。

| | V1.0 | V2.0 |
|--|------|------|
| LLM 职责 | 读数据 + 推理 + 写 Markdown | 仅从上下文提取结构化事实 |

## 四、核心功能详细设计

### 4.1 F1: Git 提交记录自动采集

**目标**：自动读取指定仓库的 Git 日志，按天聚合为结构化数据。

**数据模型**：

```python
@dataclass
class GitCommit:
    hash: str              # commit hash (短)
    author: str            # 提交者
    date: datetime         # 带时区的完整时间戳
    message: str           # commit message
    files_changed: list[str]  # 变更文件列表（仅路径，不含 diff）
    insertions: int        # 新增行数
    deletions: int         # 删除行数
```

**采集策略**：

```
输入参数：
  --repo /path/to/repo          # Git 仓库路径（支持多仓库）
  --branch-strategy merged      # 分支策略：merged | all | current
  --author "zhangsan"           # 过滤指定作者（默认当前 git config user）
  --since 2026-04-28            # 起始日期（默认本周一）
  --until 2026-05-04            # 结束日期（默认今天）

输出格式：logs/git/{repo_alias}/2026-04-28.md
```

**时区处理（V2.0 新增）**：

```yaml
# config.yaml
git:
  timezone: "Asia/Shanghai"     # 采集时使用的时区
  # Git committer date 按此 timezone 转换为本地日期
  # 确保"当天"的 commit 不会因 UTC 偏移被归到前一天
```

**分支策略（V2.0 新增）**：

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| `merged`（默认） | 只采集已合并到主分支的 commit，过滤 merge commit | 严谨统计，避免重复 |
| `all` | 采集所有分支的 commit，通过 `--no-merges` 过滤 merge | 全面但可能重复 |
| `current` | 只采集当前分支 | 快速查看 |

**Git 统计缓存（V2.0 新增）**：

```
问题：大仓库中对每个 commit 调 git diff --stat 非常缓慢
解决：SQLite 缓存 commit hash → 统计结果

data/
├── cache/
│   └── git_stats.db    # SQLite 缓存
│       # 表：commit_stats(hash TEXT PK, repo TEXT, insertions INT, deletions INT, files TEXT)
```

采集流程：
1. 读取 commit 列表（`git log`）
2. 对每个 commit，先查 SQLite 缓存
3. 缓存命中 → 直接用；缓存未命中 → 执行 `git diff --stat`，写入缓存
4. 按天聚合，生成 Markdown

**多仓库目录隔离（V2.0 修订）**：

```
logs/
├── git/
│   ├── 后端服务/              # 按 repo alias 隔离
│   │   ├── 2026-04-28.md
│   │   └── 2026-04-29.md
│   ├── 前端应用/
│   │   ├── 2026-04-28.md
│   │   └── 2026-04-29.md
│   └── _merged/              # 按天聚合的合并视图（Agent 读取这个）
│       ├── 2026-04-28.md
│       └── 2026-04-29.md
├── docs/
├── manual/
└── index/
```

`_merged/` 是自动维护的按天合并视图，Agent 读取时无需关心多仓库细节。

**commit message 质量降级（V2.0 新增）**：

```
问题：大量 commit message 是 "fix" "update"，LLM 无法从中提取有意义信息

策略：
1. 检测低质量 message（长度 < 10 字符，或匹配 /^(fix|update|wip|test)$/i）
2. 在日志中标注 [低信息量 commit]，不强行编造
3. 提示用户通过 wkr log --manual 补充上下文
4. Agent prompt 中明确："如果 commit 信息不足以推断工作内容，标注为待补充，勿编造"
```

**采集本地 commit 说明（V2.0 新增）**：

`wkr log --from-git` 采集的是**本地仓库的所有 commit**，不要求先 push。GitPython 直接读取本地 `.git`，与远程仓库无关。

### 4.2 F2: 用户文档摄入

**目标**：读取用户本地的 .md / .docx / .txt 文档，纳入周报上下文。

**摄入方式**：

| 方式 | 说明 | 命令 |
|------|------|------|
| 指定目录扫描 | 扫描目录下所有文档 | `ingest --dir ~/notes/` |
| 单文件导入 | 导入指定文件 | `ingest --file report.md` |
| 管道输入 | 从 stdin 读取 | `echo "今天做了..." \| ingest` |
| 手动录入 | 交互式输入 | `log --type manual` |

**文档解析**：

```
.md   → 直接读取 UTF-8 文本
.docx → python-docx 提取段落文本
.txt  → 直接读取
.pdf  → PyMuPDF 提取文本（可选）
```

**日志文件 Chunk 策略（V2.1 修订：含子 chunk 拆分）**：

```
问题 1：通用 chunk 策略会将同一天的日志拆到不同 chunk，破坏语义完整性
问题 2：单日内容可能过大（多仓库 + 大量手动记录），超过 embedding 模型或 LLM 的输入限制

解决：按天分 chunk + 超大日志自动子 chunk 拆分
```

```python
# chunk 策略伪代码

DAILY_CHUNK_THRESHOLD = 1500  # tokens，超过此阈值的日志需拆分

def chunk_daily_log(date, content, source, repo=None):
    """
    日志 chunk 策略：
    - 小于阈值：整日为一个 chunk
    - 超过阈值：按仓库/来源拆为子 chunk
    """
    token_count = estimate_tokens(content)

    if token_count <= DAILY_CHUNK_THRESHOLD:
        # 整日一个 chunk
        return [Chunk(
            text=content,
            metadata={"date": date, "source": source, "repo": repo, "part": "1/1"}
        )]
    else:
        # 按仓库/来源拆分
        sections = split_by_repo(content)  # 如 "后端服务: ..." 和 "前端应用: ..."
        chunks = []
        for i, section in enumerate(sections, 1):
            chunks.append(Chunk(
                text=section,
                metadata={"date": date, "source": source, "repo": repo,
                          "part": f"{i}/{len(sections)}"}
            ))
        return chunks
```

**chunk metadata 结构**：

```
{
  "date": "2026-04-28",
  "source": "git",              # git | manual | doc
  "repo": "后端服务",            # 来源仓库（可选）
  "part": "1/2",                # 同一天的第几个片段
  "indexed_at": "2026-04-28T17:00:00"
}
```

**RAG 召回去重规则**：
- 同一天的多个子 chunk 若都被召回，只保留得分最高的 1 条
- 避免某天因内容多而占据过多召回槽位

**非日志类文档**：
- 仍使用通用 chunk 策略：chunk_size=300 字符, overlap=50 字符
- chunk metadata: `{filename, chunk_index}`

### 4.3 F3: Agent 工具设计（V2.0 重大修订）

**核心原则：工具负责数据获取，LLM 负责信息提取，模板负责格式渲染。**

**修订后工具清单**：

| 工具名 | 用途 | 返回内容 | 调用方式 |
|--------|------|----------|----------|
| `get_week_summary` | 获取本周数据摘要 | 按天聚合的统计摘要（非完整日志） | **自动**（第一步必调） |
| `get_day_detail` | 获取某天详细内容 | 指定日期的完整日志 | LLM 按需调用 |
| `search_history` | 语义搜索历史 | 相关历史日志片段 | **自动**（摘要获取后自动触发） |
| `get_file_change_summary` | 获取文件变更摘要 | 文件名 + 增删行统计（**不含代码内容**） | LLM 按需调用 |
| `submit_report_data` | 提交结构化周报数据 | 校验结果 | LLM 最后一步调用 |

**与 V1.0 的关键差异**：

| | V1.0 | V2.0 | 原因 |
|--|------|------|------|
| `get_week_logs` | 返回完整日志全文 | 改为 `get_week_summary`，返回摘要 | 避免 token 爆炸 |
| `search_history` | LLM 自主决定是否调用 | **自动触发** | 避免 LLM 从不调用或每次都调用 |
| `get_git_diff` | 返回代码 diff | **移除**，替换为 `get_file_change_summary` | 与隐私策略对齐 |
| `write_report` | 接收 Markdown 保存 | 改为 `submit_report_data`，接收 JSON | 配合结构化输出流程 |

**`get_week_summary` 返回格式**：

```
本周概览（2026-04-28 至 2026-05-04）：
- 记录天数：5 天
- Git 提交：23 次
- 涉及仓库：后端服务、前端应用
- 代码变更：+1,247 / -413

按天摘要：
[周一 04-28] 后端服务: 5 commits — 主要涉及用户认证模块（login.py, auth.py）
[周二 04-29] 后端服务: 3 commits — 首页性能优化（home.py, cache.py）
           前端应用: 2 commits — 登录页面 UI 调整
[周三 04-30] 后端服务: 8 commits — API 文档 + 测试覆盖
...
```

**`search_history` 自动触发逻辑（V2.1 修订）**：

V2.0 的关键词提取方案需要额外 NLP 依赖且对中文技术日志效果差。V2.1 改为**整体 embedding 检索**：把本周日志整体编码为向量，与历史 chunk 做相似度搜索。

```python
# 在数据预处理阶段自动执行，不依赖 LLM 决策
def prepare_context(week_logs, week_summary, llm_context_window):
    # 1. RAG 自动检索：用本周日志整体作为 query（无需关键词提取）
    history_context = rag_pipeline.search_by_similarity(
        query_text=week_logs,          # 整体 embedding，语义完整
        exclude_week=week_start,
        top_k=5,
        max_chars_per_result=500       # 每条历史截断到 500 字符
    )

    # 2. 按 token 预算组装上下文（见下方详细说明）
    context = assemble_with_budget(
        summary=week_summary,
        logs=week_logs,
        history=history_context,
        llm_context_window=llm_context_window
    )
    return context
```

**上下文 Token 预算控制（V2.1 新增）**：

```
问题：预处理层可能组装出超长文本，超出 LLM 上下文窗口
      降级模式更危险：单次调用必须把全部数据塞进一个 prompt

解决：token 预算分配机制
```

```python
def assemble_with_budget(summary, logs, history, llm_context_window, mode="react"):
    """
    按优先级分配 token 预算，确保不超出 LLM 上下文窗口

    预算分配（按优先级递减）：
      1. 系统 prompt + 输出预留（固定）
      2. 数据摘要（必留，通常很短）
      3. 详细日志（优先最新一天，其余摘要化或截断）
      4. 历史上下文（固定 top_k 条，每条截断）
    """
    # 计算可用预算
    system_overhead = 2000 if mode == "react" else 3000  # 降级模式 prompt 更长
    output_reserve = 4000   # 预留给 LLM 输出
    total_budget = llm_context_window - system_overhead - output_reserve

    # 按优先级分配
    summary_budget = min(len(summary), int(total_budget * 0.1))   # 摘要：最多 10%
    history_budget = int(total_budget * 0.2)                       # 历史：最多 20%
    logs_budget = total_budget - summary_budget - history_budget    # 详细日志：剩余

    # 详细日志：优先保留最新一天，其余按时间倒序截断
    log_sections = split_by_day(logs)
    assembled_logs = ""
    for day_log in log_sections:  # 已按日期倒序排列
        remaining = logs_budget - len(assembled_logs)
        if remaining <= 0:
            break
        if len(day_log) <= remaining:
            assembled_logs += day_log
        else:
            # 截断：保留前 N 条提交，标注"共 M 条，以下为前 N 条"
            assembled_logs += truncate_day_log(day_log, remaining)

    # 历史上下文：每条截断到固定长度
    assembled_history = ""
    for item in history:
        remaining = history_budget - len(assembled_history)
        if remaining <= 0:
            break
        assembled_history += truncate(item, min(500, remaining)) + "\n"

    return f"""
=== 本周数据摘要 ===
{summary[:summary_budget]}

=== 本周详细日志 ===
{assembled_logs}

=== 相关历史背景 ===
{assembled_history}
"""
```

**降级模式压缩策略**：

```
降级模式下上下文预算更紧张（单次调用），额外压缩措施：
- 详细日志：每天只保留 commit message 列表，去掉文件名和统计
- 历史上下文：top_k 从 5 降到 3，每条从 500 字符降到 300
- 如果仍超限：按天丢弃最旧的日志，保留最新 3 天
```

## 七、定时调度方案（V2.0 修订）

### 7.1 问题分析

V1.0 使用 APScheduler，但 APScheduler 是进程内定时器，需要长驻后台进程。CLI 工具执行完就退出，无法持有 scheduler。

### 7.2 V2.0 方案：CLI 用系统 Cron，Web 用 APScheduler

```
┌─────────────────────────────────────────────────────────────┐
│                    调度方案对比                               │
├──────────────┬──────────────────┬───────────────────────────┤
│              │  CLI 模式         │  Web 模式（Phase 4）       │
├──────────────┼──────────────────┼───────────────────────────┤
│ 调度引擎      │  系统 crontab     │  APScheduler（进程内）     │
│ 前提条件      │  无               │  Web 服务长驻运行          │
│ 动态管理      │  修改 crontab     │  API 动态增删              │
│ 可靠性        │  系统级保证       │  依赖进程存活              │
│ 跨平台        │  Linux/macOS 原生 │  全平台                    │
│              │  Windows 用 schtasks│                          │
└──────────────┴──────────────────┴───────────────────────────┘
```

        Windows schtasks 不支持标准 cron，需转换为 Windows 触发器。

        支持的 cron 模式：
          "0 17 * * 5"  → /sc weekly /d FRI /st 17:00
          "30 9 * * 1-5" → /sc weekly /d MON,TUE,WED,THU,FRI /st 09:30
          "0 9 * * 1"   → /sc weekly /d MON /st 09:00

        不支持的 cron 字段（L, W, # 等）会报错并提示用户改用简单表达式。
        """
        parsed = self._parse_cron_for_windows(cron_expr)
        if parsed is None:
            raise UnsupportedCronError(
                f"Windows schtasks 不支持 cron 表达式 '{cron_expr}'。"
                "请使用简单模式，如 '0 17 * * 5'（每周五 17:00）。"
                "高级 cron 语法（L, W, #）在 Windows 上不可用。"
            )
        days, time = parsed
        cmd = (
            f'schtasks /create /tn "WkrWeeklyReport" '
            f'/tr "wkr report --config {config_path}" '
            f'/sc weekly /d {days} /st {time} /f'
        )
        subprocess.run(cmd, shell=True, check=True)

    def _parse_cron_for_windows(self, cron_expr: str) -> tuple[str, str] | None:
        """
        将 5 字段 cron 表达式解析为 (days, time) 元组
        返回 None 表示不支持的模式
        """
        DAY_MAP = {
            "0": "SUN", "1": "MON", "2": "TUE", "3": "WED",
            "4": "THU", "5": "FRI", "6": "SAT"
        }
        parts = cron_expr.strip().split()
        if len(parts) != 5:
            return None

        minute, hour, dom, month, dow = parts

        # 只支持具体时间，不支持 * 或 / 间隔
        if minute == "*" or hour == "*":
            return None
        # dom 和 month 必须是 *（不支持特定日期/月份）
        if dom != "*" or month != "*":
            return None
        # dow 不能含 L/W/#
        if any(c in dow for c in "LW#"):
            return None

        time_str = f"{hour.zfill(2)}:{minute.zfill(2)}"

        # 解析 dow
        if dow == "*":
            days = "MON,TUE,WED,THU,FRI"
        elif "-" in dow:
            start, end = dow.split("-")
            days = ",".join(DAY_MAP[str(d)] for d in range(int(start), int(end) + 1))
        elif "," in dow:
            days = ",".join(DAY_MAP[d] for d in dow.split(","))
        else:
            days = DAY_MAP.get(dow)
            if days is None:
                return None

        return days, time_str
```

### Git 统计缓存失效处理（V2.1 新增）

```
问题：commit 被 amend 或 rebase 后，hash 变了但实际是同一批变更
      缓存中旧 hash 的记录变成孤立数据

解决：
1. 缓存不做主动清理（旧 hash 不影响正确性，新 hash 会重新计算）
2. 采集时如果 git log 返回的 hash 不在缓存中 → 视为缓存未命中 → 重新统计
3. 孤立缓存条目可在 wkr cache --gc 时清理（比对 git log 中实际存在的 hash）
4. SQLite 文件本身很小（每条 ~200 字节），万级 commit 也就几 MB，不清理也可接受
```

```python
# agent/react_agent.py

class ReportAgent:
    def __init__(self, llm_config):
        self.mode = self._detect_mode(llm_config)

    def _detect_mode(self, config):
        """检测 LLM 是否支持 Function Calling"""
        if config.provider in ("openai", "dashscope"):
            # 云端 API 通常支持 FC
            return "react"
        elif config.provider == "ollama":
            # 尝试一次 FC 调用，失败则降级
            try:
                self._test_fc()
                return "react"
            except FCNotSupportedError:
                print("⚠️ 本地模型不支持 Function Calling，切换为降级模式")
                return "fallback"
        return "fallback"

    def generate(self, context: str, template: str) -> dict:
        if self.mode == "react":
            return self._react_generate(context)
        else:
            return self._fallback_generate(context)

    def _fallback_generate(self, context: str) -> dict:
        """降级模式：单次调用，prompt 中包含完整指令和上下文"""
        prompt = f"""
{FALLBACK_SYSTEM_PROMPT}

=== 以下是本周工作数据 ===
{context}

请严格按照 JSON 格式输出周报数据。
"""
        response = self.client.chat.completions.create(
            model=self.model,
            messages=[{"role": "user", "content": prompt}],
            temperature=0.3,
            response_format={"type": "json_object"},  # 部分模型支持
        )
        raw = response.choices[0].message.content
        return self._extract_json_robust(raw)

    def _extract_json_robust(self, raw: str, max_retries: int = 2) -> dict:
        """
        V2.1: 健壮的 JSON 提取（应对本地模型输出不规范）

        提取策略（按顺序尝试）：
        1. 直接 json.loads
        2. 正则匹配最外层 { ... }
        3. 去除 markdown 代码块标记后重试
        4. 追询 LLM 要求修正格式（不依赖 FC，纯文本对话）
        """
        import json, re

        # 策略 1: 直接解析
        try:
            return json.loads(raw)
        except json.JSONDecodeError:
            pass

        # 策略 2: 正则提取最外层 JSON 对象
        match = re.search(r'\{[\s\S]*\}', raw)
        if match:
            try:
                return json.loads(match.group())
            except json.JSONDecodeError:
                pass

        # 策略 3: 去除 markdown 代码块标记
        cleaned = re.sub(r'```(?:json)?\s*', '', raw)
        cleaned = re.sub(r'```', '', cleaned).strip()
        try:
            return json.loads(cleaned)
        except json.JSONDecodeError:
            pass

        # 策略 4: 追询 LLM 修正格式（最多重试 max_retries 次）
        for attempt in range(max_retries):
            fix_prompt = f"""
你上次输出的内容不是合法 JSON。请仅回复 JSON 本身，不要包含任何其他文字。

你上次的输出：
{raw[:2000]}
"""
            fix_response = self.client.chat.completions.create(
                model=self.model,
                messages=[{"role": "user", "content": fix_prompt}],
                temperature=0.1,
            )
            fixed_raw = fix_response.choices[0].message.content
            try:
                return json.loads(fixed_raw)
            except json.JSONDecodeError:
                match = re.search(r'\{[\s\S]*\}', fixed_raw)
                if match:
                    try:
                        return json.loads(match.group())
                    except json.JSONDecodeError:
                        pass
                raw = fixed_raw  # 下次重试用新的输出

        # 所有策略失败
        raise JsonExtractionError(
            "无法从 LLM 输出中提取合法 JSON。"
            "可能原因：模型不支持结构化输出、上下文过长导致输出截断。"
            f"原始输出前 500 字符：\n{raw[:500]}"
        )
```

## 九、安全与隐私（V2.0 修订）

### 9.1 三级安全策略

| 等级 | 传给 LLM 的内容 | 周报质量 | 适用场景 |
|------|-----------------|----------|----------|
| `strict` | commit message + 文件名 | 低（信息量有限） | 涉密项目、合规要求高 |
| `balanced`（默认） | message + 文件名 + 增删行数 + 函数/类名摘要 | 中 | 大多数团队 |
| `full` | 含代码 diff 片段 | 高 | 个人项目、仅用本地模型 |

### 9.2 实现

```yaml
# config.yaml
security:
  level: balanced           # strict | balanced | full

  # 关键词过滤：包含这些词的 commit 不传给 LLM
  filter_keywords:
    - password
    - secret
    - token
    - api_key
    - credential

  # full 模式下，diff 最大字符数限制
  max_diff_chars: 500
```

### 9.3 各等级采集对比

```
commit: "feat: implement JWT refresh token mechanism"
files: ["src/auth/jwt.py", "src/auth/refresh.py"]
diff: "+120 -15 (新增 JWT 刷新逻辑，使用 HS256 签名...)"

strict:   → "feat: implement JWT refresh token mechanism | jwt.py, refresh.py"
balanced: → "feat: implement JWT refresh token mechanism | jwt.py, refresh.py | +120/-15"
full:     → 上述 + diff 摘要片段（截断到 max_diff_chars）
```

### 9.4 `full` 模式与 `get_file_change_summary` 工具的关系（V2.1 澄清）

```
V2.0 矛盾：安全策略说 full 模式可传 diff，但工具定义说"不含代码内容"

V2.1 解决：get_file_change_summary 工具新增 include_diff 参数
- strict/balanced 模式：include_diff=False（默认），只返回文件名 + 增删行数
- full 模式：include_diff=True，返回文件名 + 增删行数 + diff 摘要片段

工具行为由安全等级自动控制，用户无需手动设置参数。
```

### 9.5 `files_changed` 语义澄清（V2.1 新增）

```
data_summary.files_changed = 不重复文件总数

示例：
  周一：修改 src/a.py, src/b.py → files = {a.py, b.py}
  周二：修改 src/a.py, src/c.py → files = {a.py, b.py, c.py}
  files_changed = 3（去重后）

多仓库：跨仓库同名文件算不同文件（因为路径不同）
  后端: src/main.py 和 前端: src/main.py → files_changed += 2
```

## 十一、向量索引策略（V2.0 新增）

### 11.1 增量更新

```
问题：V1.0 使用全量重建索引，随日志增多会越来越慢

解决：增量更新
- 每次新增日志 → 只对新日志做 embedding → 追加到 FAISS 索引
- metadata.json 记录已索引的日期列表
- 仅在 --force 时全量重建
```

```python
# storage/vector_store.py

class VectorStore:
    def index_log(self, date: str, content: str, source: str, repo: str = None):
        """增量索引单条日志

        V2.1 修订：去重键从 date 改为 (date, source, repo) 组合
        避免同一天的不同来源日志（如 git + manual）被跳过
        """
        key = (date, source, repo or "")
        if key in self.indexed_keys:
            return  # 已索引，跳过
        chunks = self.chunk_strategy.chunk(date, content, source, repo)
        for chunk in chunks:
            embedding = self.model.encode(chunk.text)
            self.index.add(embedding)
            self.metadata.append(chunk.metadata)
        self.indexed_keys.add(key)
        self._save_metadata()

    def rebuild(self):
        """全量重建索引"""
        self.index = faiss.IndexFlatL2(self.dimension)
        self.indexed_keys = set()
        self.metadata = []
        for log in self.log_store.list_all():
            self.index_log(log.date, log.content, log.source, log.repo)

    def search_by_similarity(self, query_text: str, exclude_week: str = None,
                              top_k: int = 5, max_chars_per_result: int = 500):
        """V2.1: 整体 embedding 检索（替代关键词提取）"""
        query_vec = self.model.encode(query_text)
        distances, indices = self.index.search(query_vec, top_k * 2)  # 多取，后续去重

        results = []
        seen_dates = set()
        for idx, dist in zip(indices[0], distances[0]):
            meta = self.metadata[idx]
            # 跳过当前周
            if exclude_week and meta["date"].startswith(exclude_week[:8]):
                continue
            # 同一天只保留得分最高的一条（V2.1 子 chunk 去重）
            if meta["date"] in seen_dates:
                continue
            seen_dates.add(meta["date"])
            results.append({"date": meta["date"], "score": dist,
                           "text": self._get_chunk_text(idx)[:max_chars_per_result]})
            if len(results) >= top_k:
                break
        return results
```

### 11.2 日志文件 Chunk 策略

```
日志类文件（logs/git/, logs/manual/）：
  → 按天分 chunk，一天 = 一个 chunk
  → chunk metadata: {date, source, repo}
  → 确保语义完整性

非日志类文档（logs/docs/）：
  → 通用 chunk 策略：chunk_size=300, overlap=50
  → chunk metadata: {filename, chunk_index}
```

## 十六、风险与应对（V2.0 修订）

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 上下文超出 LLM 窗口 | 高 | 输出截断或报错 | token 预算控制：摘要/日志/历史按优先级分配，降级模式更激进压缩 |
| 降级模式 JSON 解析失败 | 高 | 周报生成中断 | 健壮 `_extract_json`：正则 → 去标记 → 追询修正，多级容错 |
| LLM 输出 JSON 格式错误 | 中 | 渲染失败 | 全局 Schema + 模板二次校验 + 自动重试（最多 3 次） |
| Git commit message 质量差 | 高 | 提取数据无意义 | 低质量标注 + 引导手动补充 + prompt 中禁止编造 |
| 本地 LLM 不支持 FC | 高 | ReAct 循环崩溃 | 自动检测 + 降级为单次调用模式 |
| 单日日志过大 | 中 | embedding 精度下降 / 上下文截断 | 子 chunk 拆分（按仓库/来源），RAG 召回时同日去重 |
| 大仓库 Git 统计慢 | 中 | 采集耗时过长 | SQLite 缓存，增量计算，amend/rebase 时自动重算 |
| 向量索引去重遗漏 | 低 | 同日多来源日志未索引 | 去重键改为 (date, source, repo) 组合 |
| 多仓库用户名不一致 | 中 | 漏采部分 commit | 每仓库可覆盖 author，init 时检测并提示 |
| 云端 LLM 数据泄露 | 低 | 敏感信息外传 | 三级安全策略 + 启动时显式提醒 |
| Windows cron 表达式不兼容 | 低 | 定时任务安装失败 | 自动转换 + 不支持的语法报错提示 |

## 附录 B：FAQ

**Q: 为什么不用 APScheduler 做定时？**
A: APScheduler 需要长驻进程，CLI 工具执行完就退出。CLI 模式下用系统 cron，Web 模式下才用 APScheduler。

**Q: 本地 Ollama 能用吗？**
A: 能用，但体验取决于模型大小。系统会自动检测是否支持 Function Calling，不支持时降级为单次调用模式。推荐至少 7B 参数的模型。

**Q: 安全等级怎么选？**
A: 大多数团队用 `balanced`。涉密项目用 `strict`。个人项目且仅用本地模型可选 `full`。

**Q: Git 统计缓存什么时候清理？**
A: 默认不自动清理。`wkr cache --clear` 手动清理，或 SQLite 文件本身很小（每条记录 ~200 字节），万级 commit 也就几 MB。

**Q: 多仓库的 commit 会重复统计吗？**
A: 不会。每个仓库独立采集，`_merged/` 按天合并时通过 commit hash 去重。

## 附录 C：实现注意事项（边界情况）

以下 3 个边界情况不影响方案定稿，但应在写代码时顺手处理。

### C.1 token 预算硬截断的语义丢失

**问题**：当前策略对超限日志硬截断（保留前 N 条，标注"共 M 条，以下为前 N 条"），被丢弃部分的内容彻底丢失。如果丢弃的恰好是本周核心成果，周报质量会显著下降。

**MVP 处理**：硬截断 + 标注，已可接受。

**后续优化**：丢弃前对剩余内容做一次快速 LLM 摘要（一两句概括被丢弃部分的要点），注入上下文。这是一次额外 LLM 调用，成本可控。

```python
# 优化项（Phase 2+）
def truncate_day_log(self, day_log: str, budget: int) -> str:
    sections = split_by_commit(day_log)
    kept, dropped = [], []
    char_count = 0
    for s in sections:
        if char_count + len(s) <= budget:
            kept.append(s)
            char_count += len(s)
        else:
            dropped.append(s)
    if dropped:
        # 对被丢弃内容做快速摘要
        summary = self.llm.summarize("\n".join(dropped), max_tokens=100)
        return "\n".join(kept) + f"\n（另有 {len(dropped)} 条提交被省略，摘要：{summary}）"
    return "\n".join(kept)
```

### C.2 追询修正的失败兜底

**问题**：降级模式策略 4 追询 LLM 修正 JSON 格式，如果模型输出能力极差（<3B），重试两次后仍可能失败，用户只看到异常堆栈。

**处理**：`JsonExtractionError` 附带自助建议，降低排查门槛。

```python
class JsonExtractionError(Exception):
    def __init__(self, raw_output: str):
        self.raw_output = raw_output
        super().__init__(
            "无法从 LLM 输出中提取合法 JSON。\n\n"
            "自助排查建议：\n"
            "  1. 运行 wkr report --dump-context 检查预处理输出是否正常\n"
            "  2. 如果使用本地模型，尝试切换到更大的模型（推荐 ≥7B）\n"
            "  3. 如果使用云端 API，检查 API Key 是否有效\n"
            "  4. 尝试减少本周日志量（缩短时间范围）后重试\n\n"
            f"原始输出前 500 字符：\n{raw_output[:500]}"
        )
```

### C.3 整体 embedding 检索的 query 过长

**问题**：本周日志整体作为 query 编码，如果一周日志超过 embedding 模型的 token 上限（很多 sentence-transformer 默认 512 tokens），会被静默截断，检索质量下降。

**处理**：编码前做长度检查，超限时自动截取日志摘要部分（摘要天然就是全周浓缩，语义足够用于检索）。

```python
def search_by_similarity(self, query_text: str, summary: str, ...) -> list:
    """整体 embedding 检索，超长时降级为摘要检索"""
    query_tokens = self._estimate_tokens(query_text)
    max_query_tokens = self.model.max_seq_length  # 通常 512

    if query_tokens <= max_query_tokens:
        query_vec = self.model.encode(query_text)
    else:
        # 降级：用摘要代替完整日志（摘要天然短于 512 tokens）
        print(f"⚠️ 本周日志过长（{query_tokens} tokens），改用摘要检索")
        query_vec = self.model.encode(summary)

    return self._search(query_vec, ...)
```
