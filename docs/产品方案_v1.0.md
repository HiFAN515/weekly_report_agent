# 周报自动生成 Agent — 产品方案 V1.0

---

## 一、产品定位与目标

### 1.1 一句话定义

> 一个自动采集 Git 提交记录 + 用户自定义文档，基于 LLM 智能总结，按周生成结构化工作周报的 CLI/Web 双模 Agent。

### 1.2 核心痛点

| 痛点 | 现状 | 本产品解决方式 |
|------|------|----------------|
| 周五赶周报，回忆遗漏 | 手动翻 Git log、翻聊天记录 | 自动采集 Git，按天聚合 |
| 写周报耗时 30-60min | 手动组织语言、排版 | LLM 一键生成，模板可控 |
| 个人笔记/文档散落各处 | 本地 .md/.doc 无统一管理 | 统一摄入，语义检索 |
| 周报质量不稳定 | 状态好写得好，累了就敷衍 | Agent 保证结构一致、数据完整 |

### 1.3 目标用户

- **主力用户**：软件工程师、研发团队 TL
- **扩展用户**：产品经理、设计师、数据分析师（任何需要周报 + 有 Git 或文档工作流的人）

### 1.4 成功指标

| 指标 | 目标值 |
|------|--------|
| 周报生成时间 | < 2 分钟（含用户审阅） |
| 用户手动修改率 | < 20%（生成后直接可用） |
| Git 日志覆盖率 | 100%（所有 commit 不遗漏） |
| 用户满意度 | > 4.0 / 5.0 |

## 三、核心功能详细设计

### 3.1 F1: Git 提交记录自动采集

**目标**：自动读取指定仓库的 Git 日志，按天聚合为结构化 Markdown。

**数据模型**：

```
GitCommit {
    hash: str              # commit hash (短)
    author: str            # 提交者
    date: str              # ISO 日期
    message: str           # commit message
    files_changed: list    # 变更文件列表
    insertions: int        # 新增行数
    deletions: int         # 删除行数
}
```

**采集策略**：

```
输入参数：
  --repo /path/to/repo          # Git 仓库路径（支持多仓库）
  --branch main                 # 分支（默认当前分支）
  --author "zhangsan"           # 过滤指定作者（默认当前 git config user）
  --since 2026-04-28            # 起始日期（默认本周一）
  --until 2026-05-04            # 结束日期（默认今天）

输出格式：logs/git/2026-04-28.md
```

**每日聚合模板**：

```markdown
# Git 提交记录 - 2026-04-28 (周一)

## 提交统计
- 提交次数：5 次
- 涉及文件：12 个
- 代码变更：+342 / -87

## 提交详情

### 1. `a1b2c3d` — feat: 新增用户登录模块
- **文件**: src/auth/login.py, src/auth/test_login.py
- **变更**: +120 / -15

### 2. `d4e5f6g` — fix: 修复首页加载超时
- **文件**: src/pages/home.py
- **变更**: +30 / -22

...
```

**多仓库支持**：

```yaml
# config.yaml 中配置
repositories:
  - path: ~/projects/backend
    branch: main
    alias: 后端服务
  - path: ~/projects/frontend
    branch: develop
    alias: 前端应用
  - path: ~/projects/ml-pipeline
    branch: main
    alias: ML 管线
```

### 3.2 F2: 用户文档摄入

**目标**：读取用户本地的 .md / .docx / .txt 文档，纳入周报上下文。

**摄入方式**：

| 方式 | 说明 | 命令 |
|------|------|------|
| 指定目录扫描 | 扫描目录下所有文档 | `ingest --dir ~/notes/` |
| 单文件导入 | 导入指定文件 | `ingest --file report.md` |
| 管道输入 | 从 stdin 读取 | `echo "今天做了..." \| ingest` |
| 手动录入 | 交互式输入 | `log --type manual` |

**文档解析**：

```
.md   → 直接读取 UTF-8 文本
.docx → python-docx 提取段落文本
.txt  → 直接读取
.pdf  → PyMuPDF 提取文本（可选）
```

**文件组织**：

```
logs/
├── git/                    # Git 自动采集
│   ├── 2026-04-28.md
│   └── 2026-04-29.md
├── docs/                   # 用户摄入文档
│   ├── 2026-04-28_design_review.md
│   └── 2026-04-30_meeting_notes.md
├── manual/                 # 手动录入
│   └── 2026-04-29.md
└── index/                  # FAISS 向量索引
    ├── logs.index
    └── metadata.json
```

### 3.3 F3: 智能周报生成

**这是系统的核心，采用 ReAct Agent 模式。**

**生成流程**：

```
用户触发 report 命令
       │
       ▼
┌──────────────┐
│ 1. 数据采集   │ ← 读取本周所有日志 + 文档
└──────┬───────┘
       ▼
┌──────────────┐
│ 2. RAG 检索   │ ← 检索历史相关上下文（上周遗留、项目背景）
└──────┬───────┘
       ▼
┌──────────────┐
│ 3. LLM 推理   │ ← 多轮 Function Calling，Agent 自主决策
└──────┬───────┘
       ▼
┌──────────────┐
│ 4. 模板渲染   │ ← 按用户自定义模板生成最终 Markdown
└──────┬───────┘
       ▼
┌──────────────┐
│ 5. 人工审阅   │ ← 打开编辑器 / Web 预览，用户确认或修改
└──────┬───────┘
       ▼
┌──────────────┐
│ 6. 导出/分发  │ ← 保存 .md / .docx / 复制到剪贴板
└──────────────┘
```

**Agent 工具清单**：

| 工具名 | 用途 | 调用时机 |
|--------|------|----------|
| `get_week_logs` | 获取本周所有日志 | 第一步必调 |
| `search_history` | 语义搜索历史日志 | 需要上下文时 |
| `get_git_diff` | 获取某天代码 diff 摘要 | 需要技术细节时 |
| `read_document` | 读取用户指定文档 | 用户关联了外部文档时 |
| `write_report` | 保存生成的周报 | 最后一步 |
| `send_notification` | 发送通知（邮件/钉钉/飞书） | 可选，生成后通知 |

**LLM 推理策略**：

```python
# System Prompt 核心指令
"""
你是周报生成 Agent。工作流程：
1. 调用 get_week_logs 获取原始数据
2. 分析内容，识别：项目进展、技术攻关、协作事项、问题与风险
3. 如需历史背景，调用 search_history
4. 按以下结构撰写周报...
5. 调用 write_report 保存

原则：
- 用数据说话：提及具体 commit 数、文件变更数
- 问题要有解决方案：不要只列问题
- 下周计划要具体：有明确的交付物和时间节点
- 语言正式但不冗余
"""
```

```
                         ┌─────────────────────────┐

                         │  CLI (Click) | Web (FastAPI) │
                         └───────────┬─────────────┘
                                     │

                         │     Agent 编排层          │
                         │  ReAct Loop (OpenAI FC)   │
                         │  System Prompt 管理       │

                         └───────────┬─────────────┘
                                     │
              ┌──────────────────────┼──────────────────────┐
              │                      │                      │
    ┌─────────▼─────────┐  ┌────────▼────────┐  ┌─────────▼─────────┐
    │    数据采集层       │  │   知识检索层     │  │    报告生成层       │
    │                    │  │                 │  │                   │
    │ • GitPython        │  │ • SentenceTrans │  │ • Jinja2 模板引擎  │
    │ • python-docx      │  │ • FAISS 向量库  │  │ • Markdown 渲染   │
    │ • 文件系统 I/O      │  │ • 语义检索      │  │ • DOCX 导出       │

                                     │
                         ┌───────────▼─────────────┐
                         │       持久化层            │

                         │  index/ (FAISS 索引)      │

                         └─────────────────────────┘
```

### 4.2 技术选型

| 层 | 技术 | 理由 |
|----|------|------|
| CLI 框架 | **Click** | 比 argparse 更优雅，支持子命令和自动帮助 |
| Web 框架 | **FastAPI** | 异步、自带 OpenAPI 文档、前后端分离友好 |
| LLM 调用 | **OpenAI SDK** | 兼容所有 OpenAI-compatible API（本地 Ollama、通义、DeepSeek） |
| Embedding | **sentence-transformers** | 本地运行，无 API 费用，支持中文 |
| 向量库 | **FAISS** | 轻量、本地、无需外部服务 |
| Git 操作 | **GitPython** | 比 subprocess 调 git 更 Pythonic |
| 文档解析 | **python-docx / PyMuPDF** | 覆盖主流文档格式 |
| 模板引擎 | **Jinja2** | Python 生态标准，灵活可控 |
| 定时调度 | **APScheduler** | 支持 Cron 表达式，可动态添加任务 |
| 前端（可选） | **React + Tailwind** | 组件化、快速搭建 Dashboard |

│   ├── wkr ingest --dir ~/notes/   # 扫描目录

## 十、LLM Prompt 工程

### 10.1 System Prompt（周报生成）

```
你是一个专业的工作周报生成助手。

## 任务
根据用户本周的工作日志和 Git 提交记录，生成一份结构清晰、内容准确的周报。

## 原则
1. **数据驱动**：引用具体的 commit、文件名、代码行数
2. **成果导向**：强调"做了什么"和"产出了什么"，而非"花了多少时间"
3. **问题有闭环**：列出的问题必须有解决方案或处理状态
4. **计划可执行**：下周计划要有明确的交付物
5. **语言正式简洁**：避免口语化，不要用"我觉得"、"大概"

## 输出结构
# 周报 — {year}年第{week}周（{start} - {end}）

## 一、本周工作总结
### 核心成果（3-5 条，每条一句话概括）
### 详细工作（按天展开）

## 二、问题与解决方案
（表格形式：问题 | 影响 | 方案 | 状态）

## 三、下周计划
（编号列表，每项有明确交付物）

## 四、数据摘要
（提交数、代码变更量、记录天数）

## 工具使用规则
- 必须先调用 get_week_logs 获取原始数据
- 如需历史背景，调用 search_history
- 内容完整后调用 write_report 保存
```

## 十一、安全与隐私

| 风险 | 措施 |
|------|------|
| Git 代码泄露到 LLM | 只传 commit message 和文件名，不传代码 diff 内容 |
| API Key 硬编码 | 使用环境变量 + `.env` 文件，gitignore |
| 本地数据泄露 | 所有数据存本地，不上传第三方（除非用户主动配置云端 LLM） |
| 敏感信息过滤 | 可选配置：过滤含 password/secret/token 的 commit |
| 多人仓库隐私 | `--author` 过滤，只采集自己的提交 |

## 十四、风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| LLM 生成质量不稳定 | 中 | 周报质量波动 | Few-shot + 低温度 + 模板约束 |
